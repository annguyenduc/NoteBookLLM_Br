# TÓM TẮT TRI THỨC KỸ THUẬT
- **Chủ đề chính:** Gỡ lỗi Script Bash/Python, Công cụ Debug Linux, Quy trình Troubleshooting & Best Practices Git/GitHub.
- **Vấn đề / Câu hỏi:** 
  - Script nhắc nhở cuộc họp chạy không ổn định (intermittently failing) do định dạng dữ liệu đầu vào không nhất quán.
  - Phân biệt công cụ theo dõi system calls, library calls và network traffic trong Linux.
  - Xác định các bước chuẩn trong quy trình giải quyết sự cố & quản lý mã nguồn phân tán với Git.
- **Cách giải quyết / Logic lõi:** 
  **1. Chuẩn hóa đầu vào & Xử lý lỗi Script:**
  ```bash
  # meeting_reminder.sh (Sửa lỗi định dạng ngày)
  meeting_info=$(zenity --forms --title 'Meeting' --text 'Reminder' \
      --add-calendar 'Date' --add-entry 'Title' --add-entry 'Emails' \
      --forms-date-format='%Y-%m-%d' 2>/dev/null)
  [[ -n "$meeting_info" ]] && python3 send_reminders.py "$meeting_info"
  ```
  ```python
  # send_reminders.py (Parse tham số & Try-Except)
  def main():
      if len(sys.argv) < 2: return usage()
      try:
          date, title, emails = sys.argv[1].split('|')
          # Xử lý & gửi mail...
      except Exception as e:
          print(f"Failure to send email: {e}", file=sys.stderr)
  ```
  **2. Công cụ Debug Linux:**
  - `strace`: Theo dõi **system calls** (tương tác trực tiếp với Kernel).
  - `ltrace`: Theo dõi **dynamic library calls** (tương tác với thư viện ngoài).
  - `tcpdump` / `wireshark`: Bắt & phân tích **network traffic**.
  **3. Quy trình Troubleshooting & Git Workflow:**
  - **Troubleshooting:** Thu thập thông tin → Tìm root cause → Khắc phục → Documentation → Long-term remediation.
  - `git status`: Kiểm tra trạng thái staging, untracked files & sync với remote.
  - **Commit message:** Mô tả rõ `what` & `why`, format nhất quán, dùng keyword đóng issue (`Closes #ID`).
  - **Pull Request (PR):** Đề xuất thay đổi, kích hoạt code review/discussion trước merge. **Không** tự động resolve conflict.
  - `upstream`: Repository gốc mà từ đó fork được tạo ra.
- **Ghi chú quan trọng:** 
  - Lỗi `intermittently failing` thường bắt nguồn từ dữ liệu đầu vào không đồng nhất. Luôn ép kiểu/định dạng chuẩn ngay tại điểm nhập liệu (VD: `--forms-date-format`).
  - Tránh khai báo trùng khối `except Exception` trong Python. Ưu tiên log lỗi chi tiết ra `sys.stderr` để dễ debug.
  - PR là công cụ cộng tác & review chất lượng, không phải cơ chế merge tự động. Luôn `git pull upstream main` trước khi push để giảm thiểu conflict.
  - Documentation là bước bắt buộc cuối cùng của mọi quy trình xử lý sự cố, giúp tái sử dụng giải pháp và hỗ trợ bảo trì dài hạn.
